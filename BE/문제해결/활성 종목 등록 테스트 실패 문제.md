
## 배경
- 전체 통합 및 단위 테스트 실행시 AOP 관련 테스트가 실패하는 것을 확인하였습니다.
- 에러 출력 확인한 결과 `@Secured` 애노테이션에 의한 AOP 실행시 SecurityContext의 인증 정보가 존재하지 않아서 인가 관련 에러가 발생한 것입니다.
- 그러나 ActiveStockAspectTest 통합 테스트 클래스만을 대상으로 테스트들을 실행시 테스트가 통과되는 상태입니다.

## 원인 분석
- 단일 통합 테스트 클래스 실행시 다음과 `@Secured` 애노테이션 기반한 AOP 실행시 다음과 클래스로 실행됩니다.
	- `@Secured` 애노테이션 기반 AOP 실행시 `AuthorizationManagerBeforeMethodInterceptor` -> `DeferringObservationAuthorizationManager` -> `ObservationAuthorizationManager` -> `SecuredAuthorizationManager` -> `AuthoritiesAuthorizationManager` 순으로 실행됩니다.
	- SecurityContextStrategy에 의해서 SecurityContext 조회시 최종적으로 `AuthorizationManagerBeforeMethodInterceptor`의 `getAuthentication()` 메서드에서 수행됩니다.
	- `this.securityContextHolderStrategy.get().getContext()` 호출시 반환값으로 인증정보가 포함된 객체가 반환된것을 확인할 수 있습니다.

- 모든 통합 테스트 클래스 실행시 `@Secured` 애노테이션 기반한 AOP 실행시 다음 결과와 같이 SecurityContext 조회하면 인증 정보를 저장하고 있지 않습니다.
	- `@Secured` 애노테이션 기반 AOP 실행시 `AuthorizationManagerBeforeMethodInterceptor` 클래스의 `getAuthentication()` 메서드를 통해서 인증정보를 가져옵니다.
	- `this.securityContextHolderStrategy.get()` 실행시 `ThreadLocalSecurityContextHolderStrategy`를 반환합니다.
	- 하지만 `this.securityContextHolderStrategy.get().getContext()` 조회시 반한된 SecurityContext 객체 안에는 인증정보가 저장되어 있지 않은 것을 볼수 있습니다.

## 원인
### 단일 통합 테스트 클래스를 실행하는 경우에는 SecurityContext에 왜 인증정보가 저장되어 있는가?
- 단일 통합 테스트 클래스(ActiveStockAspectTest)의 Spring Context 구성시 `AuthorizationManagerBeforeMethodInterceptor` Spring Bean이 초기화되고 `@BeforeEach` 단계에서 접근하는 `ThreadLocalSecurityContextHolderStrategy`가 생성하는 ThreadLocal이 서로 같은 객체를 바라보고 있기 때문에 인증정보가 저장되어 있습니다.

### 모든 통합 테스트 클래스들을 실행하는 경우에는 SpringContext에 왜 인증정보가 저장되지 않는가?
- 앞선 통합 테스트 클래스가 실행될때 `AuthorizationManagerBeforeMethodInterceptor` Spring Bean이 최초 생성됩니다. 이때 인터셉터는 필드의 `ThreadLocalSecurityContextHolderStrategy` 객체(인스턴스 A)를 내부 필드에 고정하여 가집니다.
- 테스트 준비 단계(새로운 인스턴스 생성) : `ActiveStockAspectTest` 실행 순서가 되었을때 Spring 테스트 프레임워크가 Security 환경을 초기화/리셋하는 과정에서 Strategy 객체를 새로운 `ThreadLocalSecurityContextHolderStrategy` 객체(인스턴스 B)로 갱신하여 교체합니다.
- `@BeforeEach` 단계 (인스턴스 B에 저장) : `SecurityContextHolder.getContext().setAuthentication(auth)`은 현재 활성화된 전략 객체인 인스턴스 B의 `ThreadLocal` 에 저장됩니다.
- `portfolioService` 호출 단계 (인스턴스 A 조회 - 실패 지점) : `@Secured` 프록시를 타고 인터셉터 내부로 진입합니다. 인터셉터는 해당 인터셉터 객체가 생성될때 초기화한 인스턴스 A의 `ThreadLocal`을 참조하여 인증정보를 조회합니다.

### 결론
- 인터셉터(`AuthorizationManagerBeforeMethodInterceptor`)가 참조하는 인스턴스 A와 테스트 코드가 값을 넣은 인스턴스 B는 동일한 `ThreadLocalSecurityContextHolderStrategy` 타입이지만, 물리적으로 다른 `ThreadLocal` 인스턴스이므로 데이터가 공유되지 않습니다(null 반환).
- 단일 통합 테스트 클래스로 실행할때는 동일한 전략 객체로 인한 같은 ThreadLocal 객체를 참조하여 인증정보를 참조할 수 있었지만, 전체 테스트 시에는 Spring Bean 캐싱으로 인한 인스턴스 파편화가 발생해서 확정적으로 실패합니다.


### 모든 통합 테스트 클래스 실행시 `ThreadLocalSecurityContextHolderStrategy` 인스턴스가 새로 생성된다는 것을 검증하기

```java
@Before("@annotation(marker)")  
public void markBeforeController(JoinPoint joinPoint, ActiveStockMarker marker) {  
    try {  
       EvaluationContext context = getContext(joinPoint);  
       Object evaluatedValue = parser.parseExpression(marker.resourceId()).getValue(context);  
  
       if (evaluatedValue == null) {  
          throw new IllegalArgumentException("Evaluated resourceId is null, expression: " + marker.resourceId());  
       }  
       Set<String> tickers = new HashSet<>();  
       try {  
          tickers = switch (marker.type()) {  
             case MEMBER -> portfolioService.getAllPortfolioTickers((Long)evaluatedValue);  
             case PORTFOLIO -> portfolioService.getTickerSymbolsInPortfolio((Long)evaluatedValue);  
             case STOCK -> Set.of((String)evaluatedValue);  
             case STOCK_TARGET_PRICE ->  
                stockTargetPriceService.getAllStockTargetPriceTickers((Long)evaluatedValue);  
             case WATCHLIST -> watchListService.getAllWatchListTickers((Long)evaluatedValue);  
          };  
       } catch (Exception e) {  
	        // 🚨 [포인트 2] 에러 발생 시점의 스레드 문맥 출력  
			Thread currentThread = Thread.currentThread();  
			SecurityContextHolderStrategy strategy = SecurityContextHolder.getContextHolderStrategy();  
			  
			System.out.println("====== 🔴 [증거 2] AOP 내부 (에러 발생 시점) ======");  
			System.out.println("스레드 이름 (Name): " + currentThread.getName());  
			System.out.println("스레드 고유 ID (ID): " + currentThread.getId());  
			System.out.println("전략 객체 주소 (Hash): " + System.identityHashCode(strategy));  
			System.out.println("조회된 인증 객체: " + strategy.getContext().getAuthentication());  
			System.out.println("====================================================");  
			throw e;
       }  
  
       if (tickers.isEmpty()) {  
          return;  
       }  
       eventPublisher.publishEvent(new StocksViewedEvent(tickers));  
    } catch (PortfolioNotFoundException e) {  
       log.warn("Portfolio not found while processing ActiveStockMarker: {}", e.getMessage());  
       throw e;  
    } catch (WatchListNotFoundException e) {  
       log.warn("WatchList not found while processing ActiveStockMarker: {}", e.getMessage());  
       throw e;  
    } catch (Exception e) {  
       log.error("Failed to process ActiveStockMarker", e);  
       throw new IllegalStateException("Failed to process ActiveStockMarker", e);  
    }  
}
```

전체 통합 테스트 클래스들을 실행시 어느 특정 테스트 클래스 순간에 `ThreadLocalSecurityContextHolderStrategy`에서 `InheritableThreadLocalSecurityContextHolderStrategy`로 변경됩니다. 그런데 `portfolioService` Spring Bean 초기화시에는 SecurityContextHolderStrategy 초기화가 `ThreadLocalSecurityContextHolderStrategy`로 주입됩니다. `@BeforeEach` 단계에서 `InheritableThreadLocalSecurityContextHolderStrategy` 객체의 `ThreadLocal` 에 인증 정보를 저장해도 실제 `portfolioService` 객체가 실행될때는 `ThreadLocalSecurityContextHolderStrategy` 객체가 관리하는 `ThreadLocal`를 참조하기 때문에 인증정보가 비어 보이는 것입니다.

그러면 무엇을 검증해야 하는가?
- 전체 통합 테스트 실행시 `portfolioService` Spring Bean의 해시코드를 확인하여 변경되지 않는 것을 확인하기
- `@BeforeEach` 단계에서는 SecurityContextHolder의 `SecurityContextHolderStrategy`의 타입이 `InheritableThreadLocalSecurityContextHolderStrategy` 인 상태이기 때문에 인증 정보를 담고 있는 것을 위 테스트를 통해서 확인함
- 원인 분석 단계에서 전체 통합 테스트 클래스들 실행시 `portfolioService` 객체에서 실행하는 `AuthorizationManagerBeforeMethodInterceptor` 객체가 관리하는 `SecurityContextHolderStrategy` 타입이 `ThreadLocalSecurityContextHolderStrategy` 인것을 확인함
- portfolioService Spring Bean이 변경되지 않는 것을 확인함

그러면 어기서 궁금한것
- 어느 기준으로 `SecurityContextHolderStrategy` 의 타입이 `ThreadLocalSecurityContextHolderStrategy`에서 `InheritableThreadLocalSecurityContextHolderStrategy`으로 변경되었는지 모르겠음

### 전체 통합 테스트 실행시 `ThreadLocalSecurityContextHolderStrategy` 에서 `InheritableThreadLocalSecurityContextHolderStrategy`로 변경되는 기준 원인
통합 테스트 클래스중에 하나인 `NotificationPreferenceChangeEventListenerTest` 클래스에서 `SecurityContextHolderStrategy`를 변경하는 코드를 발견하였습니다.
```java
class NotificationPreferenceChangeEventListenerTest extends AbstractContainerBaseTest {  
  
    // ...
  
    @BeforeEach  
    void setUp() {  
       // ...
  
       // 비동기 스레드에도 SeucirtyContext 전파 설정함  
       SecurityContextHolder.setStrategyName(SecurityContextHolder.MODE_INHERITABLETHREADLOCAL);  
       setAuthentication(this.member);  
    }
	// ...
}
```

위 통합 테스트 클래스 이후에 다음 통합 테스트 클래스의 Strategy 클래스 타입을 확인한 결과 `InheritableThreadLocalSecurityContextHolderStrategy`로 변경된 것을 확인할 수 있었습니다. 다음 결과를 보면 문제의 원인이 되는 `NotificationPreferenceChangeEventListenerTest` 통합 테스트 클래스 이후에 다음 통합 테스트 클래스인 `MemberPasswordTest` 통합 테스트 클래스의 로그를 보면 SecurityContextHolderStrategy(strategy)가 `InheritableThreadLocalSecurityContextHolderStrategy`로 변경된 것을 볼수 있습니다.

### 원인 결론
1. 전체 통합 테스트 클래스들 실행시 `NotificationPreferenceChangeEventListenerTest` 통합 클래스의 `SecurityContextHolderStrategy` 를  명시적으로 기본값인 `ThreadLocalSecurityContextHolderStrategy`에서 `InheritableThreadLocalSecurityContextHolderStrategy`로 변경된다.
2.  `portfolioService` Spring Bean은 인스턴스 생성시 전체 테스트 동안에 변경되지 않았고, `portfolioService`의 프록시 객체인 `AuthorizationManagerBeforeMethodInterceptor` 인스턴스에 주입된 `SecurityContextHolderStrategy` 객체는 통합 테스트 처음 시작시 기본값인 `ThreadLocalSecurityContextHolderStrategy`로 주입됩니다.
3. 통합 테스트가 실패하는 `ActiveStockAspectTest`에서 `SecurityContextHolder.getContext().setAuthentication(auth)` 코드를 이용해서 `SecurityContext`에 인증 정보를 주입해도 해당 실행 시점에서는 변경된 `InheritableThreadLocalSecurityContextHolderStrategy` 인스턴스가 생성하는 `SecurityContext`에 인증정보를 주입하게 됩니다.
