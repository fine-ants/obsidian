
## 배경
- 전체 통합 및 단위 테스트 실행시 AOP 관련 테스트가 실패하는 것을 확인하였습니다.
- 에러 출력 확인한 결과 `@Secured` 애노테이션에 의한 AOP 실행시 SecurityContext의 인증 정보가 존재하지 않아서 인가 관련 에러가 발생한 것입니다.
- 그러나 ActiveStockAspectTest 통합 테스트 클래스만을 대상으로 테스트들을 실행시 테스트가 통과되는 상태입니다.

## 원인 분석
### `@Secured 애노테이션 수행 과정 분석`
`@Secured` 애노테이션이 추가된 메서드를 실행하면 해당 객체는 프록시 객체가 되어서 내부적으로 SecurityContext를 확인하여 인증정보를 확인하고 해당 사용자가 애노테이션에 설정된 역할을 가지고 있는지 확인합니다.

예를 들어 `portfolioService`라는 객체가 `createPortfolio()`라는 메서드를 실행합니다.

### AuthorizationManagerBeforeMethodInterceptor 실행 과정 분석
`attemptAuthorization` 메서드
- `@PreAuthorize`이나 `@Secured` 애노테이션을 처리하는 프록시 객체입니다.
- 해당 프록시 객체를 호출시 `AuthorizationManager`에게 인증 정보와 메서드 호출 정보를 전달하며 인가 검사를 위임합니다.
- 인증 정보(`Authentication` 타입) 전달시 바로 전달하지 않고 `Supplier<Authentication>`로 전달합니다.
![](refImg/Pasted%20image%2020260713110826.png)

`getAuthentication` 메서드
- 해당 메서드는 `SecurityContextHolderStrategy` 객체를 통해서 `SecurityContext` 객체를 참조하고 해당 객체안에 들어있는 `Authentication` 객체를 참조를 시도합니다.
- 만약 참조한 `Authentication` 객체가 `null`이라면 `AuthenticationCredentialsNotFoundException` 예외가 발생합니다.
- 현재 문제에서 테스트가 실패한 이유는 이 부분에서 `Authentication` 객체가 `null`로 반환되어 예외가 발생했기 때문입니다.
	- 모든 통합 테스트 클래스를 실행한후 실패한 테스트를 대상으로 디버깅한 결과
	- `this.securityContextHOlderStrategy.get()` 실행한 결과로 `ThreadLocalSecurityContextHolderStrategy` 타입의 객체를 반환함
	- `ThreadLocalSecurityContextHolderStrategy` 객체안에 존재하는 `SecurityContext` 객체에게 `Authentication` 객체를 저장하고 있는지 요청했지만 `null`을 반환함
![](refImg/Pasted%20image%2020260710123649.png)

### 통합 테스트의 SecurityContextHolder의 인증 정보 저장 분석
- 모든 통합 테스트 클래스를 수행하고 `ActiveStockAspectTest`  통합 테스트 수행 도중에 `SecurityContextHolder` 를 이용하여 인증정보를 설정합니다.
- `SecurityContextHolder.getContext()` 호출시 `SecurityContextHolderStrategy`  객체에 위임하여 `SecurityContext()`를 참조합니다.
![](refImg/Pasted%20image%2020260713113340.png)

### 단일 테스트 클래스와 모든 테스트 클래스 수행시 `SecurityContextHolderStrategy` 객체 타입의 변화 분석
인증정보를 저장하는 `SecurityContext`를 참조하는 `SecurityContextHolderStrategy` 타입이 다음과 같은 경우에 변환되는 것을 확인하였습니다.
- `ActiveStockAspectTest` 통합 클래스만을 대상으로 테스트를 수행하면 `@BeforeEach` 단계에서 `SecurityContextHolder`의 `SecurityContextHolderStrategy` 타입은 `ThreadLocalSecurityContextHolderStrategy` 이고 `AuthorizationManagerBeforeMethodInterceptor` 단계에서 `SecurityContextHolderStrategy`  필드의 타입은 `ThreadLocalSecurityContextHolderStrategy` 타입 입니다. 
- 다른 모든 통합 테스트 클래스를 수행하고 `ActiveStockAspectTest` 테스트 수행시 `@BeforeEach` 단계에서 `SecurityContextHolder`의 `SecurityContextHolderStrategy` 타입은 `InheritableSecurityContextHolderStrategy` 타입입니다. 그리고 `AuthorizationManagerBeforeMethodInterceptor` 단계에서 `SecurityContextHolderStrategy` 필드의 타입은 `ThreadLocalSecurityContextHolderStrategy` 타입입니다.

위 설명을 표로 정리하면 다음과 같습니다. 다음 표를 보면서 의아한 점은 모든 클래스 테스트 수행할때 strategy 타입이 서로 일치하지 않는다는 점입니다.
![](refImg/Pasted%20image%2020260713115638.png)
### 왜 단일 테스트 클래스(ActiveStockAspectTest) 실행시 SecurityContext에 인증정보가 저장되어 있는가?
- `@BeforeEach` 단계와 인터셉터 단계에서 strategy 객체에게 요청하여 인증정보를 참조하게 되는데, 두 단계에서 참조하는 `strategy` 객체가 동일하기 때문입니다.
- `strategy` 객체가 동일하기 때문에 `@BeforeEach` 단계에서 인증정보를 저장했기 때문에 인터셉터 단계에서도 인증정보를 참조할 수 있게 됩니다.

### 왜 모든 통합 테스트 클래스들을 실행하는 경우에는 `SpringContext`에 인증정보가 저장되어 있지 않는가?
- `@BeforeEach` 단계에서 `SecurityContextHolder`를 통해서 인증정보를 저장시 `InheritableSecurityContextHolderStrategy` 객체를 통해서 저장합니다.
- 인터셉터 단계에서 인증정보를 조회할때는 `ThreadLocalSecurityContextHolderStrategy` 객체에게 요청하여 인증정보를 참조하기 때문입니다.

 
## 원인
### 모든 통합 테스트 클래스 실행 도중에 `SecurityContextHolderStrategy`를 변경하는 코드가 존재함
통합 테스트를 수행하는 테스트 코드 중에서 `SecurityContextHolderStrategy`를 런타임에 변경하는 코드를 탐색하였습니다. 탐색한 결과 `NotificationPreferenceChangeEventListenerTest` 의 `@BeforeEach` 단계에서 strategy를 변경하는 것을 확인하였습니다. 다음 코드는 `strategy` 를 `InheritableSecurityContextHolderStratey` 타입으로 변경하는 코드입니다.
![](refImg/Pasted%20image%2020260713124412.png)

위와 같이 런타임 시점에 `strategy`가 변경되기 때문에 그 다음 통합 클래스로 실행되는 테스트들에 영향을 미칩니다. 그래서 `ActiveStockAspectTest` 실행시 `SecurityContextHolder` 클래스를 통해서 인증정보 저장시 `InheritableSecurityContextHolderStrategy` 타입의 객체를 통해서 저장합니다. 하지만 정작 인터셉터 단계에서 `strategy` 객체 접근시 `ThreadLocalSecurityContextHolderStrategy`에게 요청했기 때문에 인증정보가 `null`로 반환되는 것입니다.

`ThreadLocalSecurityContextHolderStrategy` 객체가 인증정보를 저장하지 않은 이유는 테스트가 끝나면 `SecurityContextTestExecutionListener`에 의해서 SecurityContext가 청소되기 때문입니다.

즉, 테스트가 실패한 핵심적인 원인은 모든 통합 클래스 실행 도중에 **`SecurityContextHolderStrategy`를 변경**하는 일이 있었고, `SecurityContextHolder`를 통해서 `@BeforeEach` 단계에 아무리 인증정보를 설정해도 해당 인증정보는 `InheritableSecurityContextStrategy` 타입의 객체를 통해서 저장됩니다. 그리고 **실제 인터셉터 수행시에는 Spring Context 초기화시 설정된 타입인 `ThreadLocalSecurityContextStrategy`를 통해서 인증정보를 조회를 시도했기 때문**에 인증정보를 찾을 수 없는 것입니다. 이전 테스트에서 `ThreadLocalSecurityContextStrategy` 객체를 통해서 인증정보를 저장했다 하더라도 **매 테스트가 끝나면 `SecurityContextTestExecutionListener`에 의해서 SecurityContext 정보가 청소**됩니다.
